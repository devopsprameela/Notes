Replication Rules in Amazon S3 (in Amazon Web Services)

S3 Replication rules automatically copy objects from one S3 bucket to another bucket (same or different region) for backup, compliance, or faster access.

This feature is called S3 Replication.

Types of S3 Replication Rules
1. SRR (Same-Region Replication)
Copies data within the same AWS region
Example: Mumbai → Mumbai
Used for:
Backup
Data processing separation
Compliance
2. CRR (Cross-Region Replication)
Copies data to a different region
Example: Mumbai → Singapore / US
Used for:
Disaster recovery
Low latency access for global users
Regulatory compliance
How S3 Replication Works
Source Bucket (Region A)
        ↓
Replication Rule
        ↓
Destination Bucket (Region B or same region)

Once configured:

New objects are automatically copied
Optionally, existing objects can also be replicated
Key Features of Replication Rules
1. Automatic Copying
No manual intervention needed
Works in real-time or near real-time
2. Versioning Required
Must enable versioning on both source and destination buckets
Each object version is replicated
3. IAM Role Required
AWS uses a replication IAM role
Grants permission to read source and write destination
4. Filtering Rules

You can replicate:

Entire bucket
Specific prefix (e.g., logs/)
Specific tags (e.g., env=prod)
5. Ownership Control
You can change object ownership in destination bucket
Types of Replication Configurations
1. One-way replication
Source → Destination only
2. Bidirectional replication (advanced)
Bucket A ↔ Bucket B
Used for active-active systems

S3 Bucket Policies in Amazon S3 (in Amazon Web Services)

A Bucket Policy is a JSON-based access control policy used to define who can access an S3 bucket and what actions they can perform.

It is applied directly at the bucket level.

What is a Bucket Policy?

A bucket policy controls permissions like:

Read objects
Upload objects
Delete objects
Access from specific IPs or accounts

It is similar to IAM policy but applied to a specific S3 bucket.

Structure of a Bucket Policy

A basic policy has:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
Key Components
1. Effect
Allow or Deny
2. Principal

Who can access:

"*" → public access
AWS account ID
IAM user or role
3. Action

What operations are allowed:

s3:GetObject → read
s3:PutObject → upload
s3:DeleteObject → delete
s3:ListBucket → list files
4. Resource

Which bucket or object:

Bucket: arn:aws:s3:::my-bucket
Objects: arn:aws:s3:::my-bucket/*
